🤔 Chapter 1: The Translation Problem

Imagine you only speak English, but you need to give instructions to someone who only speaks... computer. That's the fundamental problem of software development.

What You Think:

Human Language "I want a button that, when clicked, shows a welcome message"

What Computer Needs:

JavaScript document.getElementById('btn') .addEventListener('click', function() { alert('Welcome!'); } );

💡 The Real Truth

Computers are incredibly fast but incredibly stupid. They can only follow EXACT instructions, step by step, with zero understanding of context or intent. That's why we need translators - and that's what MultiAgent Claude is.

🤖 Chapter 3: The Two-Tier Agent System (Managers and Workers)

The system uses a clever two-tier hierarchy - just like a real company! Orchestrators (managers) coordinate the work, while Specialists (workers) do the actual research and planning.

🔑 Critical Concept: Research-Only Agents

Agents NEVER write code directly! They only create detailed plans. The main Claude (you're talking to) reads these plans and does the actual implementation. This prevents chaos and keeps everything organized.

The Hierarchy Explained:

🎭 Orchestrators (Opus Model)
Big-picture thinkers
Expensive but smart
🔧 Specialists (Sonnet Model)
Domain experts
Efficient and focused

How They Actually Work Together:

Complex Task (Multiple Domains)

Orchestrator Flow You: "Build a shopping cart" ↓ Master Orchestrator: "I need 3 specialists" ↓ Deploy: [Frontend, Backend, Testing] ↓ Each creates a plan file: - frontend-cart-plan-2024.md - backend-api-plan-2024.md - testing-suite-plan-2024.md ↓ Master consolidates plans ↓ Claude implements all plans

Simple Task (Single Domain)

Direct Specialist You: "Fix this CSS layout" ↓ Frontend Specialist: "Analyzing..." ↓ Creates: css-fix-plan-2024.md ↓ Claude reads plan ↓ Claude implements fix

Meet the Orchestrators (The Managers):

🎯 Master Orchestrator TOP BOSS (Opus)

Role: Analyzes your request, decides which agents to deploy

When used: Complex multi-part projects

Special power: Can coordinate up to 10+ specialists simultaneously

🌊 Wave Execution Orchestrator PHASED BOSS (Opus)

Role: Runs 7-phase systematic execution (like building a house foundation-first)

Phases: Analysis → Architecture → Integration → Testing → Documentation → Review → Deployment

Special power: Ensures nothing is missed, everything happens in right order

⚡ Parallel Controller SPEED BOSS (Opus)

Role: Runs multiple specialists at the same time

When used: When tasks don't depend on each other

Example: Frontend and backend can be built simultaneously

Meet the Specialists (The Workers):

🎨 Frontend UI Expert SPECIALIST (Sonnet)

Expertise: React, Vue, CSS, responsive design, accessibility

Creates plans for: User interfaces, components, styling

Output: Detailed UI implementation plan with component structure

🗄️ AWS Backend Architect SPECIALIST (Sonnet)

Expertise: Databases, APIs, Lambda, DynamoDB, S3

Creates plans for: Server architecture, data models, API endpoints

Output: Complete backend architecture blueprint

🧪 Playwright Test Engineer SPECIALIST (Sonnet)

Expertise: E2E testing, visual regression, test automation

Creates plans for: Test suites, CI/CD pipelines, quality checks

Output: Comprehensive testing strategy document

💡 Why This Matters to You

  • Orchestrators are expensive (Opus model) - Only used when really needed
  • Specialists are efficient (Sonnet model) - Do most of the work
  • Plans, not code - Agents write instructions, Claude implements them
  • You stay in control - Can review every plan before execution

🛠️ Chapter 5: The MAC CLI Tool (Your Control Center)

The mac command-line tool is how you set up and control your AI agent team. Think of it as the control panel for your software factory.

📝 What's a CLI?

CLI = Command Line Interface. It's typing commands instead of clicking buttons. Like texting instructions to your computer. Scary at first, but actually simpler than hunting through menus!

Setting Up Your Project (One Command!):

Terminal Command mac init This single command: ✓ Analyzes your project ✓ Sets up agent team ✓ Creates memory system ✓ Configures testing ✓ Adds CI/CD if needed

🔄 Chapter 6: The Complete Request Flow (Research → Plan → Execute)

Here's the ACTUAL process when you ask for something - it's more sophisticated than you might think!

🔑 The Three-Phase Pattern

1. Research Phase: Agents study your codebase and gather information
2. Planning Phase: Agents create detailed implementation plans
3. Execution Phase: Claude (main) reads plans and writes actual code

Example: "Build me a shopping cart with Stripe payments"

Phase 1: Research (Agents Investigate)

What Happens Master Orchestrator activates... ├─ Reads: context_session_[id].md ├─ Checks: .ai/memory/ for patterns ├─ Deploys: 3 specialist agents │ ├─ Frontend Specialist researches: │ ├─ Existing UI components │ ├─ Current styling approach │ └─ React vs Vue vs vanilla JS? │ ├─ Backend Architect researches: │ ├─ Database structure │ ├─ API patterns in use │ └─ Authentication method │ └─ Payment Specialist researches: ├─ Stripe documentation ├─ Security requirements └─ Compliance needs

Phase 2: Planning (Agents Write Instructions)

Plan Files Created .claude/doc/ ├─ frontend-cart-ui-plan-20240901.md │ ├─ Component structure │ ├─ State management approach │ ├─ UI flow diagrams │ └─ 150+ lines of details │ ├─ backend-cart-api-plan-20240901.md │ ├─ Database schema │ ├─ API endpoints │ ├─ Session handling │ └─ 200+ lines of specs │ └─ payment-stripe-plan-20240901.md ├─ Integration steps ├─ Security measures ├─ Test card numbers └─ 180+ lines of config

Phase 3: Execution (Claude Implements)

Actual Implementation Claude reads all plans and: 1. Creates React components 2. Sets up database tables 3. Writes API endpoints 4. Integrates Stripe SDK 5. Adds error handling 6. Creates test suite 7. Updates documentation RESULT: Working shopping cart!

The Actual Files Created:

Complete File Structure .claude/ ├── tasks/ │ └── context_session_12345_cart.md # Session memory ├── doc/ │ ├── frontend-cart-ui-plan-*.md # UI blueprint │ ├── backend-cart-api-plan-*.md # API design │ └── payment-stripe-plan-*.md # Payment flow └── orchestration/ └── config.json # Agent assignments .ai/memory/ ├── patterns/ │ └── shopping-carts.md # Reusable patterns └── decisions/ └── ADR-001-stripe-vs-paypal.md # Why Stripe? /src/ ├── components/ │ ├── Cart.jsx # Main cart UI │ ├── CartItem.jsx # Individual items │ └── CheckoutForm.jsx # Payment form ├── api/ │ ├── cart.js # Cart endpoints │ └── payment.js # Stripe handler └── database/ └── schema.sql # Table definitions

💡 Why This Separation Matters

  • Quality: Agents can focus on planning without worrying about syntax
  • Review: You can check plans before any code is written
  • Learning: Successful patterns get saved for next time
  • Control: You can modify plans before execution
  • No chaos: One brain (Claude) does implementation, avoiding conflicts

🚀 Chapter 8: Talking to Your AI Team

The secret to success: Clear, specific instructions with context.

❌ Vague Request:

Poor "Make me a website" Result: Generic template that doesn't fit your needs

✅ Clear Request:

Good "Build a portfolio website for a photographer with: - Grid gallery layout - Contact form - About page - Mobile responsive - Dark theme option" Result: Exactly what you need

Magic Phrases That Help:

  • "Similar to [example] but with..." - Gives AI a reference point
  • "Must include..." - Ensures critical features aren't missed
  • "For users who..." - Clarifies the target audience
  • "The main goal is..." - Helps prioritize features
  • "Don't include..." - Prevents unwanted complexity
